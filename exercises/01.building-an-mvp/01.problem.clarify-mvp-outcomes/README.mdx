# Clarify MVP Outcomes

👨‍💼 We need an MVP that helps people coordinate a shared time quickly. Before
anyone writes implementation code, we need to be clear on the outcome we are
trying to produce.

🐨 In this step, do not change application code. Write your discovery artifact
in <InlineFile file="discovery-brief.md" />. It should include:

1. Problem clarity (who, pain, and scope boundaries)
2. Success clarity (what measurable outcome makes this MVP viable)
3. Constraint clarity (time, dependencies, data/privacy, and technical limits)
4. Risk clarity (top risks and how you would validate or mitigate them)
5. Unknowns that still need answers before implementation

At minimum, include at least one concrete item for each category.

In real MVP work, start with a small, high-leverage stakeholder set:

- 💼 Berry the Business Owner (business goal and success metric owner)
- 👤 Una the User (workflow reality and usability pain points)
- Optional fourth stakeholder only if needed (legal/compliance or support/ops)

In the next step, you will run a collaborative, iterative discovery conversation.
For now, focus on preparing for that meeting and documenting what you need to
learn.

💰 Helpful prompt format:

- What decision does this answer unlock?
- What risk appears if this is wrong?
- How would we validate this in the MVP?

📜 Reference your own planning notes while drafting. Keep your assumptions,
constraints, and success criteria in one place so you can validate them in
later steps.
