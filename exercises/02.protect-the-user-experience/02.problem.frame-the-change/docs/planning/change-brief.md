# Change Brief

This brief should frame a proposed product change against the UX invariants
before implementation begins.

## Change request

- What is the requested change?
- What user problem is it trying to solve?

## Affected flows and routes

- Which core flows does this change touch?
- Which routes are involved?
- Which flows should remain unchanged?

## UX invariants at risk

- Which existing invariants matter most for this change?
- What parts of the experience must stay fast, simple, and clear?
- Where could this request accidentally introduce ambiguity or friction?

## Narrow first release

- What is the smallest version of the change worth shipping first?
- What should the first release help users do faster or more confidently?
- What should it avoid adding right now?

## Explicitly out of scope

- What related ideas should be deferred?
- What would make this change too heavy for a first release?

## Review and degradation signals

- What signs would tell you the change improved the experience?
- What signs would tell you it made the product worse?
- What should reviewers check before approving implementation?
